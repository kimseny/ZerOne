# Week 3 — 파일도 보내고 싶다: 분기 지옥의 시작

텍스트만 보내던 채팅에 **파일 전송**을 추가합니다. 메시지에 '종류'가 생기면서
같은 `if/elif` 분기가 **보내기·받기·서버 곳곳에 퍼지는** 경험을 합니다.
이 고통이 다음 주(객체/다형성)의 동기입니다 — 오늘은 일부러 해결하지 않습니다.

## 파일
| 파일 | 설명 |
|------|------|
| `server.py` | 종류(TEXT/FILE)별 `if/elif` 로 처리·중계 |
| `client.py` | 보내기·받기 양쪽에 종류별 분기. 파일은 `downloads/` 에 저장 |
| `cat.png` | 전송 실습용 샘플 이미지 (작은 빨강 PNG) |
| `recv_cut_demo/` | (곁다리) `recv` 로 받으면 큰 파일이 왜 깨지는지 보여주는 **고장난 버전** + 실습 안내 |
| `Week03_파일전송_분기지옥.pptx` | 완성된 강의 슬라이드 (발표자 노트 포함) |

## 실행 방법
```bash
# 터미널 1 — 서버
python server.py

# 터미널 2, 3 — 사람마다 하나씩
python client.py
```

대화 중:
- 그냥 입력 → 텍스트 메시지
- `/file cat.png` → 그 파일을 전송 (받는 사람의 `downloads/` 폴더에 저장됨)

## 이번 주의 약속(프로토콜)
```
클라이언트 → 서버 :  TEXT|<메시지>            |  FILE|<파일이름>|<base64>
서버 → 클라이언트 :  TEXT|<닉네임>|<메시지>   |  FILE|<닉네임>|<파일이름>|<base64>
한 메시지 = 한 줄(끝에 \n)
```
- **base64**: 바이너리(사진)를 안전한 문자열로 포장하는 변환
- **줄 단위 읽기(`makefile`)**: `recv(1024)` 하나로는 긴 base64 가 잘리므로,
  `\n` 까지 모아 읽도록 했다 (이것도 새로 늘어난 부담)
  → 잘리는 걸 직접 보고 싶으면: `recv_cut_demo/` 폴더 참고 (고장난 버전으로 재현)

## 왜 '고통'인가
종류를 하나 더 늘리면(예: 위치 공유 `LOC|`) 고쳐야 할 곳:
1. 클라이언트 **보내기** 분기
2. 서버 **처리** 분기
3. 클라이언트 **받기** 분기
4. + 프레이밍·포맷 차이 같은 잔손질

## 실습 / 과제
1. 위치 공유 `LOC|위도|경도` 추가하고 **몇 군데 고쳤는지 세어 오기** (← 핵심)
2. 같은 이름 파일은 덮어쓰지 않게 번호 붙여 저장
3. 내가 보낸 파일은 내 화면엔 저장 안 하기(보낸 사람 제외)

> 다음 주: 이 흩어진 `if/elif` 를 **메시지 객체 + 다형성**으로 통째로 없앱니다.
